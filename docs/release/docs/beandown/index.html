Write the content directly. Import the folded result as content. Reach for a normal .bst file when the job stops being content-shaped.

Beandown files

A .bd file is Markdown that lives inside Beanstalk's template world.

Write prose directly


    # Getting started

    This is **bold** text.

    [p: This paragraph is styled by the @html builder helper.]

No imports, no declarations, no frontmatter. Just content.

Import the result


    import @docs/intro {content as intro}

    [: [intro] ]

The content constant is the only thing a .bd file exports.

Remember

  • .bd is an implicit compile-time Markdown template.
  • The result is a #String constant called content.
  • No imports or declarations inside the file.

A .bd file is a Beandown content file. The compiler prepares it as an implicit compile-time Markdown template body and exposes one generated content constant.

Implicit template structure

The compiler builds each .bd file as if its whole body were the body of an implicit compile-time Markdown template:


    content #String = [$md:
        ...entire .bd file body...
    ]

The compiler constructs this structure directly from the token stream. It does not prepend wrapper source text to the file.

Exported declaration

A .bd file exposes exactly one generated constant: content #String. The content is compile-time folded HTML text. No other declaration is exported.

File is content-oriented

A .bd file is not a normal declaration file. It does not contain authored imports, declarations, frontmatter or metadata. It is a template body that produces a folded string.

Comments and punctuation

Because the file body is a template body, -- is content text, not a Beanstalk comment. Backslashes follow template-body formatter behavior. Backticks are Markdown inline-code delimiters, not raw string syntax.

Source file path and module ownership

A .bd file belongs to the module that contains its directory. It can be imported by same-module files and re-exported through the module root's export: block.

Target and builder availability

Beandown is available in HTML project builds. The .bd body sees a restricted flat compile-time scope provided by the builder. See Template scope.

Implicit Markdown

Beandown uses a small Markdown subset, not full CommonMark.

What works

  • Headings with #
  • Paragraphs separated by blank lines
  • *italic* and **bold**
  • Links
  • Unordered and ordered lists
  • Inline code with single backticks

What does not work

  • Fenced code blocks (triple backticks). Use codeblock helpers instead.
  • Pipe tables. Use structured lists instead.
  • Raw HTML is escaped. Use $raw or $html directives for raw output.

Remember

  • -- is content text, not a comment.
  • Backticks are inline-code delimiters, not raw strings.

Beandown uses Beanstalk's $md Markdown flavour. This is intentionally smaller than full CommonMark.

Supported Markdown

  • Headings: #, ##, ### and deeper
  • Paragraphs: separated by blank lines
  • Emphasis: *italic* and **bold**
  • Beanstalk-aware links: @path (label)
  • Unordered lists: - item
  • Ordered lists: 1. item
  • Inline code: paired isolated single backticks on the same line

HTML escaping

Beandown escapes HTML characters in Markdown text. Raw HTML tags such as <div> are escaped as &lt;div&gt; in the output. Use $raw or $html directives when raw HTML output is needed.

Not supported

  • Fenced code blocks (triple backtick blocks) are not rendered as <pre><code>. They appear as literal text. Use a codeblock helper with a $code(...) directive instead.
  • Markdown pipe tables are not rendered as <table>. They appear as literal text. Use a structured list or a page-owned table component instead.
  • Full CommonMark is not claimed. Beanstalk's $md is a small, deliberate subset.

Backticks are inline-code delimiters

In $md, backticks delimit inline code, not raw strings. Empty spans, repeated backtick runs, unmatched backticks, multiline spans, variable-length delimiters and fenced code blocks are not part of the $md flavour.

Links use Beanstalk-aware syntax

Use @/docs/packages (Packages) for internal routes or @https://example.com (Example) for external links. Ordinary [label](path) Markdown links belong to Plain Markdown and aren't parsed as links by Beandown's $md formatter.

-- is content

Because the file body is a template body, -- is body text, not a Beanstalk comment. It renders as literal -- in the output.

Backslashes follow body behavior

Backslashes in the template body follow formatter behavior. They are not escape characters in the same sense as normal Beanstalk source.

Code examples

Use codeblock and $code(...) where the helper is available through the content environment. See Template scope for what is visible inside a .bd file.

Content imports

Import Beandown content with normal extensionless imports.

Import and use


    import @docs/intro {content as intro}

    [: [intro] ]

The content constant is a compile-time string. Insert it wherever you want.

Do not include the extension

import @docs/intro.bd is rejected. Use import @docs/intro.

Remember

  • content is the only exported name.
  • Use as to alias it.
  • Imported content is not a page fragment by itself.

Beandown content is imported with normal extensionless source import syntax. The imported content constant is a compile-time string.

Import syntax


    import @docs/intro

    [: [intro.content] ]

A namespace import exposes content as a field. A grouped import brings it directly:


    import @docs/intro {content as intro_content}

    [: [intro_content] ]

Aliasing content

Use as to give the imported content a local name. This is the standard form when multiple content files are imported into the same page.

Compile-time string semantics

The imported content is a #String compile-time constant. It has already folded by the time the importing file uses it. It is not a runtime value.

Use in a page or template

Insert the imported content into a template body:


    import @docs/intro {content as intro}

    [: [intro] ]

The caller decides where to insert it. Imported content does not become a page fragment automatically.

Direct extension imports are rejected


    import @docs/intro.bd {content}

This is rejected with BST-IMPORT-0024. Use import @docs/intro without the .bd extension.

Relative path resolution

Content imports follow the same path resolution rules as normal source imports. @./intro resolves from the importing file's directory. @docs/intro resolves from the entry root.

Import from root or helper files

A .bd file can be imported from a module root, a normal source file, or both. The import path resolves the same way regardless of the importing file's role.

Imported content does not become a page fragment

Importing a .bd file and inserting its content into a template does not create a page fragment. The caller controls where the content appears. Only direct top-level templates in the active module root become page fragments.

Visibility and module boundary

A .bd file's content is subject to the same module visibility rules as other declarations. Cross-module imports can only access content if the module root's export: block re-exports it.


    export:
        import @./intro {
            content as intro,
        }
    ;

Template scope

A .bd file sees compile-time constants from its module root and @html helpers.

A constant capture


    -- docs/#docs.bst
    export:
        site_name #= "Beanstalk"
    ;
    

    -- docs/intro.bd
    # Welcome to [site_name]

The compile-time constant is inserted into the Markdown at that position.

A nested $raw override


    [$raw: <div>Raw HTML here</div>]

Nested templates default to $md, but an explicit directive overrides that.

Remember

  • Only compile-time constants are visible.
  • Runtime values are not visible.
  • Nested templates default to $md.

A .bd file sees a deliberately small compile-time scope. The implicit template body can access constants from its module root and builder-provided helpers.

Same-directory module root constants

Exported compile-time constants and const records from the same-directory module root are visible inside a .bd file. This is the primary way to parameterise Beandown content.


    -- docs/#docs.bst
    export:
        site_name #= "Beanstalk"
    ;
    

    -- docs/intro.bd
    # Welcome to [site_name]

Same-directory module root constants override @html constants on name collision.

Builder-provided compile-time helpers

Exported compile-time constants and const records from @html are visible flat inside a .bd file. This includes helpers such as p, codeblock and other HTML builder constants.


    -- docs/intro.bd
    [p: This paragraph uses the @html p helper.]

    [codeblock, $code("bst"): value = 42]

Nested templates

Nested templates inside a .bd body are valid. A nested template with no explicit directive defaults to $md. Any explicit directive (such as $raw, $fresh, $html, $code, $css or $escape_html) overrides the Beandown Markdown default for that nested template.


    [$raw: <div>Raw HTML output</div>]

Constants versus runtime values

Only compile-time constants are visible inside a .bd file. Runtime bindings, runtime functions, types, traits, methods, external runtime APIs and the generated self content constant are not visible.

A reference to a runtime value is rejected with BST-RULE-0034 or BST-RULE-0053.

No runtime capture

Beandown bodies are const-required. There is no runtime capture. The entire body must fold at compile time.

Allowed template directives

All template directives available in the compile-time template environment are valid inside .bd. This includes $md, $raw, $fresh, $html, $code, $css and $escape_html.

Child-template formatting boundary

Child templates are opaque barriers to the parent formatter. A child template's output is not inspected by the parent $md formatter. Template delimiters cannot pair across child boundaries.

Literal delimiter output

To output literal template delimiters, use string insertion:


    ["["]
    ["]"]
    ["[literal]"]

Content helpers

Content helpers such as codeblock are available only where genuinely provided by the builder environment. Do not assume arbitrary normal source imports are available inside .bd.

Beandown limits

Beandown is the wrong tool when the job stops being content-shaped.

Use .bst instead when

  • You need runtime values or control flow.
  • You need imports or declarations.
  • You need page fragments or slot composition.

Use .md instead when

  • You need plain Markdown with no Beanstalk syntax.
  • You need fenced code blocks or pipe tables.
  • You need raw HTML preserved.

Remember

  • .bd is const-required. Everything must fold at compile time.
  • .bd has no fenced code, no pipe tables, no raw HTML.
  • .bd sees only compile-time constants from its module root and @html.

Beandown is intentionally small. These are the confirmed restrictions and implementation gaps.

Confirmed restrictions

  • Implicit const-required evaluation: the entire .bd body must fold at compile time. There is no runtime path.
  • No normal top-level declarations: a .bd file does not contain authored imports, functions, structs, choices, type aliases or trait declarations. Text that looks like a declaration is template body content, not a compiled declaration.
  • No runtime bindings or calls: runtime values are not visible. A reference to a runtime name is rejected with BST-RULE-0034 or BST-RULE-0053.
  • No fenced code blocks: triple-backtick blocks are literal text in $md, not <pre><code>. Use codeblock helpers instead.
  • No pipe tables: Markdown pipe table syntax is literal text in $md, not <table>. Use structured lists or page-owned table components.
  • No raw backtick strings: backticks are Markdown inline-code delimiters, not raw string syntax.
  • No stored named inserts: the implementation gap for stored named inserts remains. Named inserts passed as loose contributions are rejected.
  • Template and Markdown child boundaries: child templates are opaque barriers to the parent formatter. Delimiters cannot pair across child boundaries.
  • Unsupported runtime control flow: template if and loop are available as template directives, but ordinary runtime control flow is not part of the const-required Beandown body.

Differences from normal .bst

  • A .bst file can contain imports, declarations, runtime code and page fragments. A .bd file cannot.
  • A .bst file has normal Beanstalk source scope. A .bd file has a restricted flat compile-time scope.
  • A .bst file can produce page fragments. A .bd file produces one content #String constant.

Differences from plain .md

  • A .bd file is parsed as Beanstalk template syntax. A .md file is not.
  • A .bd file can access compile-time constants and use nested templates. A .md file cannot.
  • A .bd file uses the $md renderer. A .md file uses the plain Markdown renderer (CommonMark-compatible with GFM extensions).
  • A .bd file escapes raw HTML. A .md file preserves raw HTML.

When to choose normal Templates instead

Use a .bst template when:

  • runtime values or control flow are needed
  • full slot and wrapper composition is needed
  • the content is part of broader program logic
  • imports or declarations are needed in the same file

Implementation gaps

  • Stored named inserts remain an implementation gap. See the migration ledger for details.
