MARKDOWN IMPORTS

Plain Markdown files use the .md extension for raw Markdown content fragments in HTML projects. They are helpers for .bst pages and modules, not pages themselves.

A .md file is rendered to HTML at compile time and exposed as one generated constant:


    content #String = "<rendered html>"

Use .md when the file should be plain Markdown with no Beanstalk syntax. Use @./beandown when the content needs nested templates, $markdown, or same-directory facade constants.

Importing a Markdown file

Project shape:


    src/
    ├── #page.bst
    └── docs/
        └── intro.md

Markdown source:


    # Welcome

    This is **plain Markdown**.

Beanstalk page:


    import @docs/intro

    [: [intro.content] ]

Grouped aliases work too:


    import @docs/intro {
        content as intro_html,
    }

    [: [intro_html] ]

Do not include the .md extension in the import path. import @docs/intro.md is rejected; use import @docs/intro.

What .md does not do

Plain Markdown files do not parse Beanstalk source syntax.

  • -- is Markdown text, not a Beanstalk comment.
  • $100 is text, not a template directive.
  • [not_a_template] is text, not a Beanstalk template.
  • [: body] is text, not a template body.

.md files do not support Beanstalk imports, declarations, interpolation, templates, frontmatter, metadata blocks, summaries, table-of-contents extraction, or raw-source retention. They also do not see same-directory #mod.bst constants or @html constants.

HTML, links, and assets

Raw HTML is preserved in V1. There is no sanitizer or raw-HTML policy key in this feature.

Markdown links and images render their authored paths literally:


    [Next](./next.md)
    ![Diagram](./diagram.png)

V1 does not rewrite those paths and does not copy linked or image assets. Use normal Beanstalk asset paths when a file should participate in tracked-asset output.

.md and .bd compared

Feature `.md` `.bd`
Parser CommonMark-style dependency Beanstalk `$markdown`
Beanstalk syntax No Yes, template body rules
Generated member `content #String` `content #String`
Same-directory facade constants No Yes
Frontmatter or metadata No No
Raw HTML Preserved V1 Through Beanstalk/template semantics
Links and images Literal output V1 Template-rendered output
