ERRORS AND PANICS IN BEANSTALK

Design draft: This page is a future-facing design document for language and compiler planning. It is not final user-facing documentation, and the syntax, semantics, and implementation details may change before this feature is implemented.

Errors are values in Beanstalk.

Functions that can fail return an Error value through a dedicated error return slot.

That error slot is marked with !.

Any call that can return an error must handle it.

The core rule is:

> If a function can return an error, the caller must either catch the error or bubble it upward.

Core syntax

Error-returning functions mark one return slot with !.


    parse_number |text String| -> Int, Error!:
        if text.is_empty():
            return! Error("Parse", "int.empty", "Missing number")
        ;

        return 42
    ;

In this example:

  • Int is the success value
  • Error! is the error slot
  • return! returns through the error slot
  • normal return returns through the success path

The ! marker does not mean panic.

It means this function has a typed error path that callers must handle.

The Error type

Error is a builtin language type.

It carries structured failure information instead of only a string.

Conceptually, Error contains:

Field Type Meaning
kind ErrorKind Broad category of failure
code String Stable machine-readable error code
message String Human-readable explanation
location ErrorLocation? Optional source or runtime location
trace {StackFrame}? Optional stack trace or call trace

Error, ErrorKind, ErrorLocation, and StackFrame are reserved builtin symbols.

Returning errors

Use return! to return through the error slot.


    parse_number |text String| -> Int, Error!:
        if text.is_empty():
            return! Error("Parse", "int.empty", "Missing number")
        ;

        return 42
    ;

A function with an Error! return slot can return either:

  • success values with return
  • an error value with return!

This keeps success and failure paths explicit.

Bubbling errors

Use a trailing ! to bubble an error to the surrounding function.


    wrapper |text String| -> Int, Error!:
        value = parse_number(text)!
        return value
    ;

If parse_number(text) succeeds, execution continues normally.

If it returns an error, the surrounding function returns that error from its own error slot.

The surrounding function must declare a compatible Error! return slot.

Catching errors

Use catch to inspect or recover from an error.

catch has two forms:

  • without ->, it handles the error and bubbles it upward
  • with ->, it handles the error and recovers with fallback success values

    wrapper |text String| -> Int, Error!:
        value = parse_number(text) catch |err|:
            io(err.message)
        ;

        return value
    ;

In this example:

  • err is the caught error value
  • the handler body can inspect or log the error
  • if the handler reaches the end, the same error is returned from the surrounding function

A catch block without -> must be inside a function that can return a compatible Error!, unless the block exits explicitly with return or return!.

Recovering with fallback values

Use catch -> to recover from an error with fallback success values.


    recover |text String| -> Int:
        value = parse_number(text) catch -> 0

        return value
    ;

If parse_number(text) succeeds, value receives the parsed number.

If it returns an error, value receives 0.

The fallback value must match the expected success type.

Recovering with a handler block

Use catch |err| -> when recovery needs to inspect or log the error.


    recover |text String| -> Int:
        value = parse_number(text) catch |err| -> 0:
            io(err.message)
        ;

        return value
    ;

In this example:

  • err is the caught error value
  • 0 is the fallback success value
  • the handler body can inspect or log the error
  • the surrounding function does not need to return Error!

The handler body runs only on the error path.

The fallback value is used after the handler body completes.

Overriding catch behavior

A catch block without -> automatically bubbles the caught error if the block reaches the end.

The block can override this by returning explicitly.


    recover |text String| -> Int:
        value = parse_number(text) catch |err|:
            io(err.message)
            return 0
        ;

        return value
    ;

In this example, the surrounding function returns 0 from inside the catch block.

The automatic error bubble does not happen because the block exits explicitly.

This is useful when the surrounding function does not return Error!, but the recovery path needs more control than a fallback expression.

Multiple success values

Beanstalk supports multiple return values.

A function can return multiple success values and one error slot.


    load_user |id String| -> String, Float, Error!:
        if id.is_empty():
            return! Error("User", "user.empty_id", "Missing user id")
        ;

        return "guest", 0.0
    ;

    name, score = load_user(id) catch |err| -> "guest", 0.0:
        io(err.message)
    ;

The fallback values must match the success return list.

If the success path returns two values, the recovery path must provide two fallback values.

Error handling forms

Form Meaning
parse_number(text)! Bubble the error to the surrounding function.
parse_number(text) catch -> 0 Use 0 as a fallback value if an error is returned.
parse_number(text) catch |err|: ... ; Bind the error as err, run a handler block, then bubble the same error.
parse_number(text) catch |err| -> 0: ... ; Bind the error as err, run a handler block, then recover with 0.
return! Return an error from the current function's error slot.

Collection errors

Collection operations use structured errors for invalid runtime cases.

By default, get(index) returns a result-like value that must be handled.


    my_list ~= {1, 2, 3}

    fallback = my_list.length() catch -> 0

    value = my_list.get(5) catch |err| -> fallback:
        io("Index out of bounds error: ", err.message)
    ;

If the index is out of bounds, the error value is an Error with a message field.

Collection methods should not silently ignore invalid operations.

They should produce structured errors that can be caught, bubbled, or recovered from.

Strict runtime validation

Collection methods should enforce strict runtime validation.

This includes:

  • invalid receivers
  • out-of-bounds indices
  • invalid mutation targets
  • invalid remove operations
  • backend-specific runtime failures

The backend handles propagation automatically for compiler-owned collection methods.

User code still sees the same error-handling surface:

  • recover with a fallback
  • recover with a named catch block
  • bubble the error

Options

Optional values use ?.

An optional value represents a value that may not exist.


    find_url |has_url Bool| -> String?:
        if has_url:
            return "https://nyejames.com"
        ;

        return none
    ;

    url String? = find_url(false)

none is the only special option value.

A normal value of type T can be used where T? is expected.

none requires an optional type context.

Errors vs options

Errors and options solve different problems.

Feature Use when Example
T? Absence is expected and not a failure. A user profile has no avatar.
Error! Something failed and the caller needs failure information. Parsing text failed because the format was invalid.

Use ? when missing data is ordinary.

Use Error! when failure details matter.

Panics

Panics are different from recoverable errors.

A panic means the program should not continue.

Panics are intended for cases such as:

  • runtime assertions
  • impossible states
  • critical invariant violations
  • debugging unexpected state
  • stopping before unsafe or invalid behavior could continue

Panic syntax:


    #panic "Message about the panic"

Panics should be rare.

Most expected failures should use Error! instead.

Errors vs panics

Mechanism Meaning Caller responsibility
Error! Recoverable failure returned as a value. Must catch, recover, or bubble.
T? Optional absence of a value. Must handle according to optional-value rules.
#panic Runtime stop caused by an unrecoverable condition. Should not be used for normal control flow.

Use panics for broken assumptions, not regular failures.

Full example


    parse_number |text String| -> Int, Error!:
        if text.is_empty():
            return! Error("Parse", "int.empty", "Missing number")
        ;

        return 42
    ;

    wrapper |text String| -> Int, Error!:
        value = parse_number(text)!
        return value
    ;

    log_then_bubble |text String| -> Int, Error!:
        value = parse_number(text) catch |err|:
            io(err.message)
        ;

        return value
    ;

    recover |text String| -> Int:
        value = parse_number(text) catch |err| -> 0:
            io(err.message)
        ;

        return value
    ;

    fallback_value = parse_number("") catch -> 0

This example shows the core call-site choices:

  • bubble with parse_number(text)!
  • recover with parse_number(text) catch -> 0
  • catch, inspect, and bubble with catch
  • catch, inspect, and recover with catch |err| -> 0

Diagnostics

Error handling should produce clear diagnostics.

Useful diagnostic cases include:

  • calling an error-returning function without handling the error
  • using trailing ! in a function with no compatible error return slot
  • using catch without -> in a function with no compatible error return slot, unless the block exits explicitly
  • using fallback values with the wrong type or wrong count
  • using return! in a function that does not return Error!
  • using none without optional type context
  • using #panic where only declarations are allowed

Error handling is part of the type system.

It should not be treated as a backend-only runtime behavior.

Summary

Beanstalk separates three kinds of failure or absence:

  • Error! for recoverable errors with structured information
  • ? and none for optional values
  • #panic for unrecoverable runtime stops

The core rules are:

  • functions that can fail mark one return slot with !
  • error-returning calls must be handled
  • return! returns an error
  • trailing ! bubbles an error
  • catch -> fallback recovers with fallback success values
  • catch |err|: ... ; catches, handles, and bubbles the same error
  • catch |err| -> fallback: ... ; catches, handles, and recovers
  • panics should not be used for normal control flow

The result is explicit, typed, and compiler-checkable error handling.
