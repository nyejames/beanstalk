IMPLEMENTATION MATRIX

This page tracks the current implemented compiler surface.

It is an implementation matrix, not a full language specification. The wider language docs may describe intended or aspirational syntax. This page should describe what the compiler currently supports, rejects cleanly, or keeps experimental.

Use this together with tests/cases/manifest.toml when changing language features.

Alpha target

The supported Alpha target is:

  • frontend parsing and diagnostics
  • typed AST construction
  • HIR lowering
  • borrow validation
  • JavaScript backend
  • HTML project builder
  • docs-style static site generation

A policy for the repo is all tests must pass before each commit. So all tests on all platforms can be assumed to be green for the current state of the compiler unless specifically noted here.

Module constants remain compile-time metadata rather than runtime top-level statements. Body-local # constants are supported in declaration scopes when their initializer fully folds; runtime locals and function parameters remain invalid in compile-time constant initializers. Const records are field-access-only: palette.red and fully projected nested fields such as palette.primary.label are supported, but const-record values cannot be assigned to runtime values, passed, returned, collected, or materialized as HIR runtime struct constructs. Private inferred const facts are tracked internally during AST finalization and carried into HIR as advisory metadata; config validation consumes these facts to allow references to earlier compile-time constants while rejecting runtime expressions. Borrow checking, temporary-local reduction, and backend lowering do not consume HIR const metadata yet.

Borrow validation and ownership analysis Supported Broad borrow, aliasing, mutable/exclusive access, use-after-move, branch merge, loop, short-circuit, lifetime, drop-site, and adversarial coverage. Do not make ownership semantic; GC fallback remains the language baseline.
Paths and imports Supported Broad import syntax, relative child path, entry root fallback, source library prefix, circular dependency, multi-file module, namespace imports, grouped aliases, nested grouped aliases, collision, project-structure collision, module-boundary, and file-local coverage. Frontend / HTML builder Path rendering is golden-sensitive. Keep normalized output and platform-stable diagnostics covered. Source and external package namespace imports are supported as shallow field-access-only records: value/function/constant members are accessed as namespace.member, type members are valid in type position as namespace.TypeName, and record values cannot be passed, returned, assigned, or traversed recursively. Specific imports use grouped syntax; direct symbol-path imports are rejected with BST-IMPORT-0017, explicit .bst import extensions are rejected with BST-IMPORT-0020, and explicit external file extensions without an active provider are rejected with BST-IMPORT-0021. Grouped source import aliases and nested grouped source import aliases are supported. Same-module ordinary top-level declarations are importable by default without the old # visibility prefix. Directories containing non-config #*.bst files are module boundaries; #mod.bst is the only public facade. Cross-module and source-library imports go through #mod.bst facades, and modules without a facade cannot be imported from outside. Sibling name.bst / name/ collisions are rejected during Stage 0 project discovery, while .js files are excluded from that specific collision rule. Authored declarations in #mod.bst are exported automatically, while normal imports inside #mod.bst stay local unless wrapped by authored facade declarations. Import aliases are file-local and do not create public facade names. Alias and visible-name collisions are hard errors across same-file declarations, source imports, external imports, namespace records, type aliases, prelude symbols, builtins, and facade public names. Direct imports of special files such as #mod.bst, #page.bst, and #config.bst are rejected with BST-IMPORT-0008. Prelude and external symbols cannot be redeclared as user functions or variables at top level or in body scope. Alias case mismatches warn. Grouped import aliases reject group-level trailing aliases, invalid local alias names, and duplicate alias syntax with structured diagnostics. Parent-directory imports with `..` are not supported, and imports must stay within allowed module/library/project roots.
External platform packages Partial Broad package-scoped import and visibility coverage, with targeted external constant, fallible external signature, Float/F64/Char ABI, core libraries, prelude, duplicate-name, non-imported-symbol, alias diagnostics, JS runtime-import validation, and project-local JS provider frontend coverage. Frontend / JS Functions, opaque types, and compile-time external constants are supported through Rust-side registry metadata.
  • General user-authored external binding files are not supported. The HTML builder now supports the first provider-backed external files: annotated single-file .js imports with @bst.opaque and @bst.sig comments.
  • Project-local, source-library, and builder-owned JS-backed packages can create frontend-visible package-scoped opaque types, free functions, and receiver methods. Namespace imports, grouped imports, grouped aliases for functions/types/receiver methods, receiver-call lookup, fallible Error! signatures, parser rejection for malformed this receiver parameters, provider cache deduplication, parsed registered-runtime imports, block-bodied arrow export support, expression-bodied arrow export rejection, and emitted JS runtime-asset metadata are covered.
  • Direct .js imports obey the same module/source-library privacy boundaries as Beanstalk source imports: private JS implementation files cannot be imported across #mod.bst facade boundaries. Dev-server watch scopes include direct .js files under the entry root and configured library folders while ignoring generated output.
  • The HTML JS path now generates glue modules for provider-created ES module exports, imports emitted JS assets by relative module specifier, and maps stable external function IDs to generated wrapper names. Fallible JS wrappers convert bstOk / bstErr results into the backend's internal carrier shape, debug glue rejects invalid wrapper shapes, and release glue returns an error carrier instead of treating arbitrary JS values as success.
  • Registered core JS runtime modules such as @beanstalk/runtime are emitted once per build and resolved from generated pages through an import map when accepted JS runtime imports require them. Runtime imports are parsed from the JS source, may use multiline named import formatting, validated against registered exported names, and are not inferred from fallibility. Fallible JS functions may use bstOk / bstErr helpers or return the wrapper shape manually. The HTML builder provides the first built-in JS-backed package, @web/canvas, with explicit imports, opaque Canvas / Canvas2d types, fallible canvas/context lookup helpers, and a small receiver-method drawing surface.
  • External constants are implemented for scalar compile-time values (Int, Float, Bool) and lower through expression parsing as ordinary literals. Non-scalar external constants are rejected in constant contexts.
  • External ABI metadata includes F64, I32, Bool, Char, and Utf8Str for the current core package skeletons and JS provider signatures.
  • External function metadata supports frontend-visible success slots plus an optional final concrete error slot. Fallible external calls must use postfix ! or boundary catch: / catch |err|: handling, and they lower through the same explicit fallible HIR control-flow path as source functions.
  • External custom error slots must reference registered frontend-visible external types. Unknown external signature types are rejected instead of being smuggled through T!.
  • Explicit imports are required for non-prelude external packages. io and IO are registered as external prelude symbols; builtin error symbols are compiler-owned bare names.
  • External function aliases and external constant aliases are supported. External opaque type aliases are rejected with structured diagnostics. Grouped external aliases and shallow external package namespace imports are supported.
  • Prelude collision rejection is enforced: explicit imports cannot shadow prelude names, and prelude symbols cannot be redeclared as user declarations.
  • External receiver methods are supported for explicitly imported JS surfaces and are keyed by exact package-scoped opaque receiver types when those types come from JS annotations. Keep this surface narrow; broader external-method APIs beyond @web/canvas remain future design work.
  • JS lowering now resolves external runtime names through per-function lowering metadata in the registry.
  • InlineExpression JS lowering is implemented for pure expression external functions. Arguments are substituted positionally using `#0`, `#1`, ... placeholders; duplicate placeholders are rejected. No runtime helper is emitted for inline-only functions.
  • Provider-created JS functions carry ExternalModuleExport metadata with the original ES module export name for the generated-glue phase. They are intentionally not treated as direct inline runtime helpers.
  • Wasm lowering for external packages is experimental and should not be counted as Alpha support.
  • Unsupported external package function calls for the active backend are rejected with structured BST-RULE-0058 diagnostics before backend lowering, not during LIR transformation.

Source library roots Partial Targeted coverage for AST-backed config loading, known-key enforcement, config value-shape enforcement, plain immutable and explicit compile-time config bindings, folded template settings, config-local type/struct/choice support declarations, core/builder source-library config imports, project-local and relative config import rejection, unsupported config functions/mutable bindings/standalone templates/runtime-only values, config local const references between declarations, config-defined source library root resolution, missing/non-directory explicit scan roots, prefix collision detection, empty entry roots, `.bst` file/folder collision rejection, and `#mod.bst` facade enforcement. Frontend / HTML Source libraries and regular modules share the same visibility model. Config loading now runs `#config.bst` and reachable core/builder source-library support files through tokenizer, header parsing, dependency sorting, and AST construction before extracting folded immutable known-key declarations authored in the config file. Config declarations must use known core or builder-declared keys, and each known key's registered value shape is enforced before core fields are applied or backend settings are stored. String settings accept string literals and folded templates, project is a closed string set currently accepting only "html", boolean settings require folded Bool values, and library_folders accepts one string or a collection of strings. Preferred source-library scan syntax is library_folders = {"lib", "packages"}; explicit compile-time syntax such as library_folders #= {"lib"} is also supported. Folded template initializers are accepted for string settings. Const-record field projection is supported for explicit #= config keys and deferred for plain = config keys pending shared const-context or resolver support. Authored config may also contain imports, type aliases, structs, and choices as compile-time support declarations. Prefix collisions between builder-provided and project-local libraries produce hard errors. Maintenance rule

When a language feature changes shape:

  1. Update this page.
  2. Update tests/cases/manifest.toml.
  3. Add or rewrite canonical cases instead of layering temporary coverage.
  4. Mark unsupported syntax as Deferred or Experimental, not as a vague missing test.
  5. Do not count HTML-Wasm as Alpha-supported unless the roadmap explicitly changes.
  6. When external packages change, update the package registry, backend lowering, import visibility tests, and this matrix together.

The goal is to keep the Alpha surface small, honest, and fully exercised.
