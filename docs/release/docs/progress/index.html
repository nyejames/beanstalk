IMPLEMENTATION MATRIX

This page tracks the current implemented compiler surface.

It is an implementation matrix, not a full language specification. The wider language docs may describe intended or aspirational syntax. This page should describe what the compiler currently supports, rejects cleanly, or keeps experimental.

Use this together with tests/cases/manifest.toml when changing language features.

Alpha target

The supported Alpha target is:

  • frontend parsing and diagnostics
  • typed AST construction
  • HIR lowering
  • borrow validation
  • JavaScript backend
  • HTML project builder
  • docs-style static site generation

A policy for the repo is all tests must pass before each commit. So all tests on all platforms can be assumed to be green for the current state of the compiler unless specifically noted here.

Paths and imports Supported Broad import syntax, relative child path, entry root fallback, source library prefix, circular dependency, multi-file module, namespace imports, grouped aliases, nested grouped aliases, collision, project-structure collision, module-boundary, and file-local coverage. Frontend / HTML builder Path rendering is golden-sensitive. Keep normalized output and platform-stable diagnostics covered. Source and external package namespace imports are supported as shallow field-access-only records: value/function/constant members are accessed as namespace.member, type members are valid in type position as namespace.TypeName, and record values cannot be passed, returned, assigned, or traversed recursively. Specific imports use grouped syntax; direct symbol-path imports are rejected with BST-IMPORT-0017, explicit .bst import extensions are rejected with BST-IMPORT-0020, and explicit external file extensions without an active provider are rejected with BST-IMPORT-0021. Grouped source import aliases, nested grouped source import aliases, and grouped imports for virtual external packages are supported. Grouped virtual external package imports resolve through external package metadata before source/module facade enforcement, preventing root/module facades from misclassifying imports such as import @web/canvas { get_canvas } while preserving strict direct symbol-path rejection and source facade privacy. Same-module ordinary top-level declarations are importable by default without the old # visibility prefix. Directories containing non-config #*.bst files are module boundaries; #mod.bst is the only public facade. Cross-module and source-library imports go through #mod.bst facades, and modules without a facade cannot be imported from outside. Sibling name.bst / name/ collisions are rejected during Stage 0 project discovery, while .js files are excluded from that specific collision rule. Explicit facade status: authored declarations in #mod.bst are public only when marked with export, while unmarked declarations and ordinary imports stay private to the facade file. export import @path { Symbol } and grouped export @path { Symbol } public re-exports are supported for source and external package symbols, aliases define the public facade name, duplicate public names are rejected with BST-RULE-0081, public authored facade APIs reject private type surfaces, including inferred exported constant types, with BST-RULE-0082, and bare namespace exports such as export @path remain deferred. Alias and visible-name collisions are hard errors across same-file declarations, source imports, external imports, namespace records, type aliases, prelude symbols, builtins, and facade public names. Direct imports of special files such as #mod.bst, #page.bst, and #config.bst are rejected with BST-IMPORT-0008. Prelude and external symbols cannot be redeclared as user functions or variables at top level or in body scope. Alias case mismatches warn. Grouped import aliases reject group-level trailing aliases, invalid local alias names, and duplicate alias syntax with structured diagnostics. Parent-directory imports with `..` are not supported, and imports must stay within allowed module/library/project roots.

  • External ABI metadata includes F64, I32, Bool, Char, and Utf8Str for the current core package skeletons and JS provider signatures.
  • External function metadata supports frontend-visible success slots plus an optional final concrete error slot. Fallible external calls must use postfix ! or boundary catch: / catch |err|: handling, and they lower through the same explicit fallible HIR control-flow path as source functions.
  • External custom error slots must reference registered frontend-visible external types. Unknown external signature types are rejected instead of being smuggled through T!.
  • Explicit imports are required for non-prelude external packages. io and IO are registered as external prelude symbols; builtin error symbols are compiler-owned bare names.
  • External function aliases and external constant aliases are supported. External opaque type aliases are rejected with structured diagnostics. Grouped external aliases and shallow external package namespace imports are supported.
  • Prelude collision rejection is enforced: explicit imports cannot shadow prelude names, and prelude symbols cannot be redeclared as user declarations.
  • External receiver methods are supported for explicitly imported JS surfaces and are keyed by exact package-scoped opaque receiver types when those types come from JS annotations. Keep this surface narrow; broader external-method APIs beyond @web/canvas remain future design work.
  • JS lowering now resolves external runtime names through per-function lowering metadata in the registry.
  • InlineExpression JS lowering is implemented for pure expression external functions. Arguments are substituted positionally using `#0`, `#1`, ... placeholders; duplicate placeholders are rejected. No runtime helper is emitted for inline-only functions.
  • Provider-created JS functions carry ExternalModuleExport metadata with the original ES module export name for the generated-glue phase. They are intentionally not treated as direct inline runtime helpers.
  • Wasm lowering for external packages is experimental and should not be counted as Alpha support. Reachable @web/canvas calls remain JS-backed and are rejected for HTML-Wasm with structured unsupported-backend diagnostics.
  • Unsupported external package function calls for the active backend are rejected with structured BST-RULE-0058 diagnostics before backend lowering, not during LIR transformation. Validation is reachable-call-based so unused wrappers do not fail builds for backends that would reject the call if reached.

Core time package Partial Targeted import, grouped alias, namespace, receiver-method visibility, runtime smoke, fallible parse/catch, arity/type diagnostics, old-name rejection, opaque-field rejection, reachable Wasm-unsupported, and unused-wrapper reachability coverage. JS / HTML @core/time exposes opaque Duration, TimeMark, and Timestamp types plus the first JS-backed time helper slice.
  • Monotonic elapsed-time helpers: mark_now(), elapsed_since(start), and duration_between(start, end).
  • Wall-clock timestamp helpers: timestamp_now(), timestamp_from_unix_seconds(seconds), timestamp_from_unix_milliseconds(milliseconds), and fallible timestamp_from_iso_string(text).

Maintenance rule

When a language feature changes shape:

  1. Update this page.
  2. Update tests/cases/manifest.toml.
  3. Add or rewrite canonical cases instead of layering temporary coverage.
  4. Mark unsupported syntax as Deferred or Experimental, not as a vague missing test.
  5. Do not count HTML-Wasm as Alpha-supported unless the roadmap explicitly changes.
  6. When external packages change, update the package registry, backend lowering, import visibility tests, and this matrix together.

The goal is to keep the Alpha surface small, honest, and fully exercised.
