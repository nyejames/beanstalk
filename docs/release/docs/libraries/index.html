Libraries and imports

Imports make names visible. Module roots decide what crosses a module boundary. Source libraries give reusable modules stable prefixes.

Those are separate jobs. Keeping them separate is what stops a filesystem from quietly becoming an API.

Choose the explanation level: Import paths

Import paths

Imports make names visible. Each path reaches a declaration surface.

A relative import


    import @./card {render_card}

@./card resolves to card.bst in the same directory as the importing file.

A library prefix import


    import @helper {double}

@helper resolves from a configured source-library folder. Each direct child directory of lib/ becomes a prefix.

Remember

  • @./ means same directory.
  • @../ is not supported.
  • Source imports omit the file extension.
  • JavaScript imports keep .js.

Import paths resolve declaration surfaces, not file bytes. Each path family reaches a different kind of importable target.

Path families

  • Relative current directory: @./file resolves from the importing file's directory. The file extension is omitted.
  • Relative parent directory: @../file is not supported. Imports with .. are rejected with BST-IMPORT-0016.
  • Module path: non-relative imports whose first segment does not match a source-library prefix resolve from the configured entry_root. Example: @components/card resolves to src/components/card.bst when entry_root is src.
  • Source-library prefix: a non-relative import whose first segment matches a configured source-library prefix resolves from that library root. Example: @helper {double} when lib/helper/ is a source library.
  • Core package: @core/math, @core/text, etc. are registered by the builder as external packages.
  • Builder source-library package: @html is a builder-provided source library.
  • External package: @web/canvas is a builder-owned runtime package.
  • JavaScript file: @./drawing.js imports a project-local annotated JavaScript file. The .js extension is kept.

Resolution rules

  • @./ resolves relative to the importing file.
  • .bst, .bd and .md source imports are extensionless.
  • Direct .js imports keep the .js extension.
  • Non-relative imports whose first segment matches a source-library prefix resolve from that library root.
  • Other non-relative imports resolve from the project's configured entry_root.
  • Core and builder runtime packages use their registered package path.
  • Specific symbol imports use grouped syntax. Direct symbol-path imports such as import @core/math/sin are rejected.
  • Direct imports of hash-root files and config.bst are rejected.
  • Beandown .bd and plain Markdown .md content files both expose generated content constants. See Beandown content imports and Markdown imports.
  • The imported public surface of a module root comes from its export: block. See Module visibility.

Relative imports inside a module

Files in the same module can import ordinary sibling source files directly. Same-module imports bypass the public-surface boundary. See Module visibility.

Cross-module imports

Imports that cross into another module boundary can only access what that module's export: block makes public. Cross-module imports cannot bypass the public API. See Module visibility.

Content-file imports

Beandown and plain Markdown content files are imported with the same extensionless syntax. Their import shape is covered by the Beandown and Plain Markdown pages, not here.

Choose the explanation level: Grouped and namespace imports

Grouped and namespace imports

Grouped imports bring multiple symbols. Namespace imports bring a whole surface.

Two colliding render symbols


    import @blog { render as render_blog }
    import @docs { render as render_docs }

Two modules both export render. Aliases keep them apart.

A namespace alias


    import @core/math as math

    result = math.sin(1.0)

math gives you field access to the whole math package.

Remember

  • Use as to rename individual symbols.
  • Namespace imports are field-access-only, not first-class values.
  • An import alias must not hide an existing name.

Grouped and namespace imports provide compact syntax for importing multiple symbols or a whole declaration surface.

Grouped symbol imports


    import @core/math {sin, cos, PI}

    import @components {
        render as render_component,
        Button as UiButton,
        Card,
    }

Each entry in a grouped import brings one symbol into the file. as creates a file-local alias. Grouped imports expand into individual symbol imports.

Namespace imports


    import @core/math

    angle = math.PI
    result = math.sin(angle)

A namespace import brings the whole surface as a field-access-only record. Use math.sin(1.0) for values and math.PI for constants, but do not pass the namespace record itself as a value.

Namespace alias


    import @core/math as math

    import @vendor/drawing.js as drawing

A namespace alias gives the namespace a local name. This is the standard form for external package namespaces.

Grouped symbol alias


    import @core/math {
        sin as sine,
        PI as pi,
    }

Each entry can have its own alias. There is no trailing group-level alias.

Importable symbol categories

  • functions
  • structs
  • choices
  • type aliases
  • traits
  • compile-time constants

Runtime top-level bindings are not importable. See Module visibility.

Namespaced access

Source and module namespace imports are shallow. External package namespaces can expose nested package-local symbols. Use field-access syntax on the namespace record, but do not pass the namespace itself as a first-class value. Source-authored receiver methods are called through receiver-call syntax rather than as namespace fields.

Imports are compile-time declarations

Import records are not first-class runtime values. They exist at compile time to make names visible.

Collision and no-shadowing policy

  • Ordinary import aliases are file-local.
  • An import alias cannot hide an existing visible name.
  • If render is already declared in the file, import @other { render } is an error. Pick a different local name.
  • Leading-case mismatches warn. For example, importing a struct with a lowercase alias or a function with an uppercase alias triggers a warning.

No trailing alias for a group

A grouped import cannot have a trailing group-level alias. Each entry gets its own alias individually.

Choose the explanation level: Module visibility

Module visibility

Declarations are private to their module. The export block makes them public.

An internal helper file


    -- src/components/utils.bst
    internal_value #= 42

The root module uses it directly


    -- src/#page.bst
    import @components/utils {internal_value}

Same-module imports see everything. No export block needed.

Other modules need the export block


    -- src/#page.bst
    export:
        import @components/utils {internal_value}
    ;

Now a nested module at src/pages/ can import internal_value through @components/utils.

Remember

  • Same module: all declarations are visible.
  • Cross module: only exported declarations are visible.
  • Runtime bindings are never importable.

Module visibility controls which declarations cross a module boundary. Normal declarations are private to their module. The export: block marks what other modules can see.

Same-module internal visibility

Files within the same module can import each other's declarations directly. A module is defined by its root directory. All .bst files in that directory (and its subdirectories that do not have their own root) belong to the same module.


    -- src/components/utils.bst
    internal_value #= 42
    internal_fn || -> String:
        return "internal"
    ;
    

    -- src/#page.bst (same module)
    import @components/utils {internal_value, internal_fn}

Same-module imports bypass the public-surface boundary. The importing file sees all declarations in the target file.

Cross-module public boundary

A different module can only access declarations marked public by the module root's export: block. Cross-module imports cannot bypass the public API.


    -- src/#page.bst (root module)
    export:
        import @components/utils {
            internal_value,
            internal_fn,
        }
    ;
    

    -- src/pages/#page.bst (different module)
    import @components/utils {internal_value}

The nested module can import internal_value because the root module's export block re-exports it. Without the export block, the import is rejected with BST-IMPORT-0011.

Root-private declarations

Declarations in the module root file outside the export: block are private to the root. They are not available to other modules even through the public surface.

Direct root import rejection

Direct imports of hash-root files are rejected. Import the module through its directory path, not through the root filename.

Runtime top-level bindings not importable

Runtime top-level bindings are start-body code. They are not importable declarations. Only compile-time constants, functions, structs, choices, type aliases and traits are importable.

Receiver method visibility

Receiver methods are visible through a module's public surface when the module exports the receiver type's source surface. A type alias in the export block does not automatically re-export private implementation methods. Export the receiver type's source surface or a wrapper explicitly.

Import cycles

Circular imports between files in the same module are accepted when the declarations resolve in dependency order. Circular constant dependencies are rejected with BST-RULE-0033.

No public access through filesystem tricks

There is no way to bypass the module public-surface boundary by choosing a different import path. All cross-module imports go through the same resolution and visibility check.

Choose the explanation level: Public re-exports

Public re-exports

The export block can re-export imported symbols so other modules can use them.

A public re-export


    export:
        import @components/card {
            CardData as Card,
            render_card,
        }
    ;

Other modules can now import CardData and render_card through the module's public surface.

Local alias is not public


    import @components/card {CardData as LocalCard}

LocalCard is file-local. It is not public. Only imports inside the export: block become public re-exports.

Remember

  • Re-exports go inside the export: block.
  • The as alias defines the public API name.
  • Legacy inline export syntax is rejected.

The export: block can re-export grouped imports from other files, creating a public API surface that exposes imported declarations under controlled names.

Grouped re-export


    export:
        import @components/card {
            CardData as Card,
            render_card,
        }

        import @core/math {
            PI,
            sin,
        }
    ;

The original symbol names (CardData, render_card, PI, sin) become accessible to cross-module imports. The public alias names (such as Card) are public names of the module's API surface.

Renamed public re-export

The as alias inside an export-block re-export defines the public API name. The original source name is what the module imports privately. The alias is what other modules see when importing the module's public surface.

Exported imported types, constants and functions

Any importable declaration can be re-exported: functions, structs, choices, type aliases, traits and compile-time constants.

Re-export does not make local import alias automatically public

A private import @path { Symbol as LocalName } outside the export block creates a file-local alias. It does not become public. Only an import inside the export: block creates a public re-export.

Public name collisions

Public names in the export block must not collide with each other. Duplicate public names are rejected.

Invalid unknown or private targets

Re-exporting a symbol that does not exist in the target file is rejected. Re-exporting a private declaration that is not visible in the target file is rejected.

No legacy inline export syntax

Legacy inline export syntax (without the : block delimiter) is rejected with BST-SYNTAX-0001. The strict export: block is the only public API marker. See Public API for the full export-block rules.

Choose the explanation level: Source libraries

Source libraries

Source libraries are Beanstalk source under lib/. Each direct child folder becomes an import prefix.

A simple library


    lib/
    └── blog/
        ├── #api.bst
        └── helpers/dates.bst

#api.bst is just a cosmetic root filename. Any #*.bst name works.

Importing from a library


    import @blog {post_title}

@blog resolves to lib/blog/. The library's public surface comes from its export block.

Remember

  • lib/ is the default library folder.
  • Each direct child becomes a prefix.
  • Internal files are hidden unless re-exported.

Source libraries are normal Beanstalk source roots discovered from configured library folders. Each direct child directory becomes an import prefix.

Configuration


    library_folders #= {
        "lib",
    }

When library_folders is omitted, /lib is the default scan folder.

Directory structure


    project-root/
    ├── config.bst
    ├── src/
    │   └── #page.bst
    └── lib/
        └── helper/
            ├── #api.bst
            └── utils.bst

The direct child folder lib/helper becomes the @helper library prefix. The library's public surface is controlled by its module root (#api.bst).

Generic hash-root naming

The library root filename after # has no semantic role. #api.bst, #mod.bst and #page.bst are all equivalent root spellings for a source library. See Module roots.

One root per library module

Each library directory may contain at most one #*.bst root file. Two or more hash-root files in one library directory are rejected with BST-CONFIG-0001.

Public surface

Outside importers can only see declarations and grouped re-exports marked public by the library root's export: block. Internal implementation files are not public unless the root re-exports them.

Internal child files

Files without # in the library directory are normal source files within the library module. They are hidden from cross-prefix imports unless the library root re-exports them.

Custom scan roots

library_folders accepts one string or a collection of strings. Each configured folder becomes a scan root. Its direct child directories become import prefixes.

Duplicate or ambiguous prefixes

Duplicate library folders are rejected. Absolute paths and paths containing .. are rejected. Nested paths such as lib/helpers are rejected. Each entry must be a direct folder name relative to the project root.

Source libraries compile as Beanstalk source

Source libraries are not opaque external packages. They are normal Beanstalk source files compiled through the same frontend pipeline. The project does not own the library's filesystem location in the same way it owns src/, but the compiler treats the source as ordinary Beanstalk.

API-only source-library roots

A source-library root with no builder-relevant top-level activity is API-only. It exports declarations but produces no HTML artifact. See HTML routing and artifacts.

Project library versus core, builder and external packages

Source libraries are project-local Beanstalk source. Core packages (@core/math), builder source libraries (@html) and external packages (@web/canvas) are builder-provided. See Core, builder and external packages for the distinction.

Choose the explanation level: Core, builder and external packages

Core, builder and external packages

Different packages come from different places. The builder decides what is available.

Why @html is available

The HTML builder provides @html as a builder source library. It is normal Beanstalk source bundled with the builder. Another builder may expose different packages.

Three kinds

  • Source libraries: your own Beanstalk source under lib/.
  • Core packages: typed external functions and constants from the builder, such as @core/math.
  • External packages: builder-owned runtime packages such as @web/canvas.

Remember

  • The builder selects which packages exist.
  • External packages expose free functions and opaque types only.
  • No package manager yet. Use lib/ for local libraries.

Beanstalk distinguishes compiler core source libraries, builder-provided source libraries, external packages and host packages. Each has a different provider and visibility model.

Compiler core source libraries

Core packages are typed external packages registered by the builder. They expose opaque external types, compile-time constants and external free functions. They are not .bst source files.


    import @core/math {sin, PI, clamp}

    angle = PI / 2.0
    wave = sin(angle)

Core libraries require explicit imports unless they are part of the prelude. The io namespace alias and compiler-owned Error are available without explicit imports. Prelude external symbols do not override source declarations or explicit imports.

Builder-provided source libraries

Builders can provide source libraries without requiring those files to exist in the user's project. The HTML builder provides @html.


    import @html {center}

    #[center: HOME]

Builder source libraries follow the same source-library idea as project-local libraries. The builder registers a library prefix, imports use that prefix directly, and the source files are still Beanstalk source.

External and host packages

External packages are virtual packages provided by backend metadata. The HTML builder's @web/canvas is a builder-owned JavaScript library bundled with the project builder.


    import @web/canvas {get_canvas, Canvas2d, context_2d}

External packages expose free functions and opaque types only. They do not expose receiver methods. Use free functions for raw external APIs, or a source-owned wrapper type when method-style ergonomics are wanted.

Source declarations versus opaque external values

Source library and builder source library imports provide typed Beanstalk declarations from .bst files. Core and external package imports provide opaque external values registered by the builder. The frontend can type-check both, but only source declarations have visible internal structure.

Builder-selected availability

The builder decides which packages exist. Unsupported builder packages are rejected with an unsupported-by-builder diagnostic. The HTML builder currently opts into optional core packages such as @core/math, @core/text, @core/random and @core/time. Other builders may expose a different core surface.

Target-specific reachability rejection

The HTML-Wasm mode has a stricter backend support surface. JS-backed packages need a Wasm lowering before they can be used by that backend path. The builder rejects unsupported reachable features at lowering time.

External calls are positional-only

External package free functions use positional parameters. Named arguments, default parameters and reactive parameters are source-language features not available in external signatures.

Namespace and alias rules

External aliases follow normal file-local collision and case-convention rules. Explicit external imports must not collide with visible source symbols. External opaque types can be passed, returned and used by external functions, but cannot be constructed with struct syntax or field-accessed.

Absence of package-manager semantics

There is no package manager, remote fetching, lockfiles or version resolution yet. The practical model is: use lib/<name>/#*.bst for project-local Beanstalk libraries, use @core/..., @html and @web/canvas for builder-provided libraries, and use small annotated .js wrapper files when a project needs browser JavaScript directly.

Current feature availability

The progress matrix is the source of truth for current package and backend support.
