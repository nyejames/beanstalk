PROJECTS, MODULES AND HTML BUILDER

Beanstalk projects are built around a small set of conventions:

  • a project root with a #config.bst file
  • an entry root, usually src
  • module entry files named like #page.bst
  • module API boundaries defined by #mod.bst
  • normal .bst files that provide declarations for module entries
  • builder-owned output folders such as dev and release

#mod.bst = module API / visibility boundary #page.bst = HTML builder entry/configuration convention #config.bst = build/project configuration convention

At this stage of development, this is the only supported project structure to rely on. The rules are deliberately narrow so the compiler, dev server, and HTML builder can stay predictable while the language is still evolving.

The modular build system

The Beanstalk compiler frontend is shared by every project builder. It discovers modules, parses files, type checks code, lowers to HIR, and runs borrow validation. A project builder then decides how those compiled modules become real output files.

The current user-facing project builder is the HTML project builder. It consumes each compiled module and emits one HTML page per module entry file.

This split matters because Beanstalk is not designed as a single hard-coded web compiler. The same frontend can later feed other builders, such as a Wasm module builder, a documentation builder, or project-specific tooling.

Layer Responsibility
Compiler frontend Discovers modules, resolves imports, checks types, lowers code, validates borrowing, and produces backend-ready module data.
Project builder Interprets compiled modules for a specific project type and decides which output files to generate.
HTML project builder Treats each module entry as a page and emits HTML files, with runtime code embedded as JavaScript by default.

Project root and #config.bst

A normal HTML project has a #config.bst file at the project root. Config files use ordinary Beanstalk top-level binding syntax.


    project = "html"
    entry_root = "src"
    dev_folder = "dev"
    output_folder = "release"
    page_url_style = "trailing_slash"
    redirect_index_html = true
    name = "html_project"
    version = "0.1.0"
    author = ""
    license = "MIT"

    origin = "/beanstalk"
    html_lang = "en"
    html_title_postfix = " | Beanstalk"
    library_folders = {
        "lib",
    }

Config keys are strict about value shape. String settings accept string literals and folded templates, project currently accepts only "html", library_folders accepts one string or a collection of strings, template_const_loop_iteration_limit requires a positive folded Int no greater than 1_000_000, and boolean HTML settings such as redirect_index_html require real true / false values rather than strings.

Config values must resolve to compile-time constants. Plain immutable key declarations and explicit #= key constants are accepted, and they may reference earlier compile-time config keys or constants imported from core/builder source libraries. #config.bst may contain imports, type aliases, structs, and choices as support declarations, but every authored value-producing declaration must still be a known config key. Config-local helper constants, functions, calls, runtime statements, project-local imports, relative config imports, and #[...] page fragments are rejected. Const-record field projection is supported for explicit #= config keys and deferred for plain = keys.

Important settings:

Setting Meaning
project Selects the project type. For the documentation site and current web projects, use "html".
entry_root The source folder that is searched for module entry files. Usually "src".
dev_folder Where development builds are written. The dev server serves from here.
output_folder Where release builds are written when --release is used.
library_folders Configures source-library scan roots. Each direct child directory of a scan root becomes an import prefix. `/lib` is the default when omitted.
template_const_loop_iteration_limit Sets the per-loop compile-time template expansion guard. The default is 10_000.
HTML-specific settings Settings such as origin, html_lang, html_title_postfix, page_url_style, and redirect_index_html are interpreted by the HTML builder.

Config is loaded before module discovery. That means entry_root and library_folders shape how the project is searched and how imports resolve.

Default project scaffold

The bean new html command generates a complete default project:


    project-root/
    ├── #config.bst
    ├── src/
    │   └── #page.bst
    ├── lib/
    ├── dev/
    │   └── .beanstalk_manifest
    ├── release/
    │   └── .beanstalk_manifest
    └── .gitignore

Clarifications:

  • lib/ is created empty by default. It is the conventional place for project-local source libraries.
  • dev/ and release/ are output folders, not source folders.
  • .beanstalk_manifest files are build-system metadata used for safe stale-output cleanup.
  • src/#page.bst is the default HTML page entry and includes a starter page.

Module entry files

A module entry file is a Beanstalk file whose file name starts with #. In directory projects, the compiler searches the configured entry root for #*.bst files. #config.bst is not a module entry.

Examples:


    src/#page.bst              -> module entry
    src/docs/#page.bst         -> module entry
    src/docs/basics/#page.bst  -> module entry
    src/styles/docs.bst        -> normal source file
    #config.bst                -> project config, not a module entry

Only module entry files have top-level runtime execution. The compiler treats the entry file's top-level runtime code as an implicit start function for that module.

Normal source files can contain imports, functions, structs, choices, type aliases, and constants. They should not rely on top-level runtime statements being executed by themselves.

Builder-owned meaning of # file names

The compiler discovers #*.bst files as module entries. The project builder decides what those entries mean.

The HTML project builder currently treats them as page entries. The file name and folder decide the output route.

Entry file HTML output path
src/#page.bst index.html
src/docs/#page.bst docs/index.html
src/docs/basics/#page.bst docs/basics/index.html
src/#about.bst about/index.html

This gives pages stable folder-backed routes. A #page.bst file means "the index page for this folder". Other #name.bst files become named routes under the same folder.

Top-level templates in page entries

Top-level templates are special. In the HTML project builder, direct top-level templates in a module entry file become page fragments.

There are two forms:

Form Meaning
#[...] A top-level const template. It must fold at compile time and is emitted as static HTML.
[...] A top-level runtime template. It is evaluated by the module's implicit start function and inserted into the page at runtime.

Example:


    import @styles/docs {navbar, section, theme_head}

    page_title #= "Home"
    page_description #= "Welcome page."
    page_head #= theme_head

    -- Const fragments are folded by the compiler and emitted directly.
    #[navbar]

    #[section, $markdown:
        # Welcome
        This part is static.
    ]

    name = "Beanstalk"

    -- Runtime fragments are produced by the implicit start function.
    [section, $markdown:
        Runtime hello from [name].
    ]

The HTML builder preserves source order between static and runtime fragments. Static fragments are emitted immediately. Runtime fragments use generated slots in the HTML document, then the embedded JavaScript calls the module start function and fills those slots.

Do not put page-producing top-level templates in helper files. Helper files should export reusable declarations, then page entries should import and use them.

Visibility and compile-time constants

Declaration kind Import behavior
Runtime top-level binding Entry-file start-body code only. It is not importable.
Compile-time constant Written as name #= value or name #Type = value. It is importable as a declaration.
Function Importable by other files in the same module by default.
Struct, choice, or type alias Importable by other files in the same module by default.
Top-level template #[...] is not a normal exported value. It is an entry-file-only const page fragment.

# no longer means public visibility. It marks compile-time constant bindings. Cross-module public visibility is controlled by #mod.bst facades.

Example helper file:


    -- src/components/card.bst

    CardData = |
        title String,
        body String,
    |

    render_card |card CardData| -> String:
        return [:
            <article>
                <h2>[card.title]</h2>
                <p>[card.body]</p>
            </article>
        ]
    ;

    helper_note || -> String:
        return "This stays internal unless a facade wraps it."
    ;

Example page entry:


    -- src/cards/#page.bst

    import @components/card {CardData, render_card}

    page_title #= "Cards"

    card = CardData("Hello", "This was rendered by a helper function.")

    [:
        [render_card(card)]
    ]

Module boundaries and imports

Imports are how a page entry reaches helper declarations, module facades, source libraries, core packages, and JavaScript packages. This page only covers the project-shape part of that model. For full import syntax, source libraries, core packages, JavaScript imports, and builder-provided libraries, see Libraries and modules.

Project-shape rules to remember:

  • files in the same module can import ordinary sibling source files directly
  • imports that cross into another module root must go through that module's #mod.bst facade
  • source-library prefixes come from library_folders or from the builder
  • direct imports of #mod.bst, #page.bst, and #config.bst are rejected
  • sibling name.bst / name/ import-name collisions are rejected during project discovery

In this shape:


    src/
    ├── #mod.bst
    ├── #page.bst
    ├── components/
    │   └── card.bst
    └── docs/
        └── #page.bst

src/#page.bst can import @components/card directly because they are in the same module. src/docs/#page.bst is a different module entry, so it imports the root module's public surface through src/#mod.bst instead of bypassing into src/components/card.bst.

Example: the documentation site structure

The Beanstalk documentation website is itself a Beanstalk HTML project. Ignoring markdown notes and generated files, its source shape is roughly:


    docs/
    ├── #config.bst
    ├── src/
    │   ├── #mod.bst
    │   ├── #page.bst
    │   ├── docs/
    │   │   ├── #page.bst
    │   │   ├── basics/#page.bst
    │   │   ├── templates/#page.bst
    │   │   ├── functions/#page.bst
    │   │   ├── if-statements/#page.bst
    │   │   ├── loops/#page.bst
    │   │   ├── collections/#page.bst
    │   │   ├── structs/#page.bst
    │   │   ├── choices/#page.bst
    │   │   ├── pattern-matching/#page.bst
    │   │   ├── aliases/#page.bst
    │   │   └── progress/#page.bst
    │   └── styles/
    │       └── docs.bst
    ├── lib/
    │   └── local_helpers/
    │       └── #mod.bst
    ├── assets/
    │   └── ...
    ├── dev/
    └── release/

The important part is the split:

  • src/ contains the root page, the root #mod.bst facade, and ordinary source files.
  • #page.bst files define HTML pages and make their folders module boundaries.
  • styles/docs.bst is a normal source file imported by the root facade and then selectively exposed to nested page modules with export declarations in src/#mod.bst.
  • @html is provided by the HTML builder.
  • lib/local_helpers/#mod.bst would expose a project-local @local_helpers source library.
  • assets/ is a normal project folder for static files such as images.
  • dev/ and release/ are output folders, not source folders.

Output files

The build command compiles the project, then writes the builder's output files under the selected output root.

Build mode Output root
Default / dev #dev_folder, usually dev.
--release #output_folder, usually release.

In the default JS-backed HTML mode, each page module produces an HTML file. The generated JavaScript is embedded into that HTML file.

In the experimental --html-wasm mode, the builder can instead emit a route folder containing:

  • index.html
  • page.js
  • page.wasm

The HTML builder owns stale-output cleanup for generated HTML artifacts. Generated output folders should be treated as build artifacts, not source.

Module facades at a glance

#mod.bst is the public surface for a module root. It is not a shared implementation file, and sibling files do not see its declarations automatically.

Keep facades small and intentional:

  • mark public wrapper functions, type aliases, structs, choices, traits, and constants with export
  • re-export grouped symbols with export import @path { Symbol } or export @path { Symbol }
  • keep unmarked declarations and normal imports private to the facade file
  • import private implementation details inside the facade when a wrapper needs them
  • do not put runtime top-level statements or runtime top-level templates in a facade
  • do not expect a normal import inside #mod.bst to become public by itself

    -- src/#mod.bst

    import @core/text {length as private_length}

    private_prefix #= "docs"

    export page_label #= "Documentation"

    export title_length |title String| -> Int:
        return private_length(title)
    ;

    export @components/card {
        CardData as Card,
        render_card,
    }

The same facade idea is used by project-local source libraries under lib/<name>/#mod.bst. The detailed library, facade, core package, and JavaScript import guide is Libraries and modules.

Future extension points

#test.bst test modules

A future #test.bst file can hold project tests. Normal builds would ignore it. A planned test command could discover and run those files, similar to how cargo test discovers Rust tests.

This is separate from the current compiler-internal integration test runner.

$doc documentation templates

The $doc style directive can eventually make documentation part of Beanstalk source. Top-level or nested doc templates could be stripped from normal output code, then collected by a documentation command.

This would make documentation generation feel closer to Rust doc comments, but with Beanstalk's template system instead of comment syntax.
