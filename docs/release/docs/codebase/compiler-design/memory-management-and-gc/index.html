Automatic memory management before the borrow checker

Heap values must stay usable while something can still observe them. They must also release resources without letting code touch storage that already died. Those two sentences sound like one problem. They are at least two: keeping wood alive while it still feeds the canopy, and stopping two branches from tearing the same limb.

Garbage collection answers liveness. Access validation answers conflicting use. Beanstalk runs both stories on purpose.

Five memory strategies

Compare strategies by when storage ends, not by marketing labels.

Manual allocation. The programmer calls allocate and free. Maximum control. Maximum opportunity to free too early, free twice or forget entirely.

Reference counting. Each alias bumps a count. When the count hits zero, storage frees. Cycles need special care. Costs sit on every alias edge.

Tracing GC. A collector walks from roots and reclaims what it cannot reach. Programmers stop writing most frees. Pauses and throughput become runtime concerns.

Ownership. Each value has a unique owner responsible for destruction. Transfers move that duty. Alias rules get strict. Deterministic drops become natural.

Region or arena allocation. Many values share a bulk lifetime. The region frees as a unit. Great for bursty workloads with clear epochs. Awkward when lifetimes tangle.

Real systems mix these. Beanstalk's semantic baseline is GC. Ownership-aware lowering may ride on top when proof exists.

Failure modes

Name the failure modes:

  • use after free reads or writes storage that already ended
  • double free destroys the same storage twice
  • leaks retain storage forever without intending to
  • invalid aliasing creates overlapping views the language did not allow
  • conflicting mutation lets one writer smash what readers still treat as stable
  • data races under concurrency deserve separate treatment. Label them as general or future-concurrency context rather than current Beanstalk thread syntax

GC aims at lifetime hazards. It does not automatically bless conflicting mutation. Two aliases can both point at live memory and still disagree about who may write.

What GC solves

A tracing collector (or a host GC such as the JavaScript heap) tracks reachability from roots. Unreachable objects become reclaimable.

That property underwrites Beanstalk's semantic promise: correctness does not depend on proving a deterministic drop on every path. If ownership analysis cannot prove a transfer, the value may remain GC-managed. Observable behaviour stays stable.

What ownership adds

Ownership thinking gives deterministic destruction and clearer resource release when the compiler can prove a last use. It also adds source complexity if you expose lifetimes and move keywords everywhere.

Beanstalk refuses that source tax. There is no lifetime syntax and no move keyword in the language surface this series teaches. Moves are inferred. Copies are explicit. Existing values default to shared read-only access. Mutation needs explicit exclusive access through ~place.

Beanstalk's semantic baseline

Hold these rules together:

  1. GC is the semantic baseline for all backends.
  2. Borrow validation is mandatory, even under GC.
  3. Shared access is the default for existing values.
  4. copy creates an independent value on purpose.
  5. ~place requests temporary exclusive access to a mutable place.
  6. Moves are inferred at safe consumption points.
  7. Ownership-aware lowering is optional optimisation layered over the same accepted programs.

GC-only and ownership-aware backends must accept and reject the same source programs. Optimisation may not widen or shrink the language.

Backend choices

Targets pick runtime machinery:

  • host GC on JavaScript, letting the browser or engine reclaim
  • Wasm GC proposals and engines where available
  • linear memory with handles and runtime helpers
  • hybrid paths that mix host objects and Wasm-managed storage

Those choices implement one semantic contract. They do not redefine shared access or exclusivity. Ownership optimisation stays subordinate to language semantics.

In the greeting project, strings and template values cross runtime operations. Under GC they remain safe while reachable. Under ownership-aware Wasm lowering they might drop earlier when proof exists. Priya's paragraph still means the same thing.

What can fail here

  • unsafe aliasing remains invalid even when GC preserves allocation lifetime
  • conservative ownership evidence may retain values under GC longer than a perfect oracle would
  • do not imply deterministic destruction as a correctness requirement for source programs

Why Beanstalk does it this way

Alternative: expose Rust-style lifetimes, references and owned-or-borrowed signatures in every API.

Choice: keep GC as the semantic fallback and validate access separately.

Benefit: safer source rules with a backend escape route when proof is thin.

Cost: collector work remains, and ownership-aware destruction cannot always prove a transfer.

Roadmap and current status

  • Accepted design: GC is the semantic baseline for all backends.
  • Current implementation: memory and progress authorities for runtime coverage.
  • Queued work: ownership-aware backend work follows mixed backend plans.
  • Deferred or outside scope: current concurrency syntax, and deterministic destruction as a source-level guarantee.
