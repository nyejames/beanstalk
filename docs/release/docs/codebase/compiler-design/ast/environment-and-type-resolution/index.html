Environment construction

build_ast_environment consumes header-built file visibility, then resolves:

  • declaration metadata
  • compile-time constants
  • nominal structs
  • choices
  • function signatures
  • receiver catalog data
  • trait metadata
  • shared environment side channels

AST resolves declaration shells in sorted order. It may register nominal identity and generic parameter metadata before constants so constructors are name-resolvable. Field and variant constructor shells stay unresolved until their semantic TypeIds can be checked and written to TypeEnvironment.

Type resolution

type_resolution/ owns parsed type-reference resolution to canonical TypeId.

It handles:

  • source-visible lookup
  • type aliases
  • builtin types
  • options
  • growable collections
  • fixed-capacity collections
  • hashmaps
  • generic parameters
  • concrete generic nominal instantiation
  • external package types that have frontend mappings

Fixed capacity is type identity. {Int} and {4 Int} are different semantic types. Maps are their own constructed shape. They are not collection variants.

Visibility through ScopeContext

AST consumes the header-built file visibility environment through ScopeContext. It can validate semantic use of visible symbols, but it must not rebuild import bindings or rediscover top-level visibility.

Expression and type resolution should use the active ScopeContext, not global bare-name lookup.

All user-visible names go through one collision policy. Same-file declarations, source imports, external imports, type aliases, prelude symbols and builtins can't silently shadow each other.

Public facade API validation

module_ast/environment/public_surface.rs owns semantic public facade API validation after type and trait identities are resolved.

Public facade APIs must not expose private facade-only types in:

  • signatures
  • fields
  • aliases
  • generic bounds
  • exported constant types
  • exported trait metadata

Receiver methods are visible through a facade when the facade exposes the receiver type's source surface. Type aliases in #mod.bst don't automatically re-export private implementation methods.

TypeEnvironment boundary

TypeEnvironment is the canonical module-local source of semantic type identity. TypeId equality in the active environment is the only semantic comparison rule.

Function signatures store canonical TypeIds on ReturnSlot and Declaration after resolution. Parallel DataType values remain available for diagnostics only.

AST body emission receives AstTypeInterner, a narrow facade over TypeEnvironment. It allows derived type interning and module-local compatibility caching without permitting nominal declaration mutation.

Member lookup

TypeEnvironment member queries expose borrowed field/variant views and direct member lookup helpers. AST, HIR and backend-facing lowering should use those semantic views instead of cloning member lists for lookup.

Source fields, receiver calls and compiler-owned collection/map builtin member access live under field_access/.

Public API drift hazards

Be strict here. Subtle type-resolution drift spreads everywhere.

Common mistakes:

  • using DataType for semantic comparisons
  • resolving names through global maps instead of ScopeContext
  • rebuilding import visibility in AST
  • cloning member lists for lookup instead of using semantic views
  • leaking private facade-only types through public API surfaces
  • treating receiver methods as independently importable symbols
  • resolving fixed collection capacity as a backend allocation hint
  • modelling maps as collection capacity variants
