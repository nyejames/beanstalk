Packaging a module once

Later stages need a coherent object, not a loose mutable AST and a prayer. One shared module must serve every entry that needs it without recompiling once per page.

A successful module compilation produces an immutable CompiledModuleArtifact, a sealed growth ring the rest of the build can graft onto without cutting the trunk open again.

Four artefact lanes

Think of four layers in one stem, plus rings that record what changed:

Public semantic interface. What consumers may observe: exported identities, canonical types, folded constants, generic templates, traits, receiver surfaces, access and effect summaries, project-context provenance.

Executable state. Module-local TypeEnvironment, validated HIR and borrow-analysis facts.

Link facts. Backend-neutral per-function edges, binding-backed call ids, helpers, capabilities, reactive features, numeric and cast operations, paths, assets, provenance and generated-function requests.

Compiler and builder metadata. Dormant root activity, folded fragment values and insertion indexes, entry metadata, documentation fragments, path usages, structured warnings.

Fingerprints sit beside the lanes so reuse can say what changed.

Backend planning facts do not live in the public interface. Private bodies do not leak into consumer-visible identity.

For greeting.bst, the constant and function land in the interface when exported, in executable state as HIR and types, and in link facts as call and capability edges as needed.

Fingerprints and identity

Accepted design records separate fingerprints, including:

  • public-interface fingerprint
  • implementation fingerprint
  • dormant root-activity fingerprint
  • runtime-dependency fingerprint
  • documentation fingerprint

Invalidation meaning follows the split:

  • a private body change need not recompile semantic consumers unless a public fact changes
  • an implementation change may force relinking or codegen without semantic consumer recompilation
  • a root-activity change relinks entries that activate the module
  • a runtime-dependency change updates capability, glue and asset planning
  • a documentation-only change regenerates docs without invalidating executable instances

Stable origin identities and export bindings keep re-exports honest. Renaming a re-export alias changes the exporting module's public fingerprint. It does not rewrite the origin declaration's identity.

Success, diagnosed and blocked

Outcomes stay strict:

  • Success yields a full artefact with no errors inside
  • Diagnosed yields diagnostics and no partial public interface
  • blocked consumers waiting on a required diagnosed provider do not pretend to link
  • CompilerError aborts the owning compilation boundary

A diagnosed provider cannot expose "half of greet" for experimental linking. Partial linkable data is a bug farm.

Generated sidecars

Base module artefacts remain immutable. Concrete generic functions live in generated sidecars owned by the consuming project or package compilation.

Each sidecar carries its request identity, type environment story, concrete HIR, borrow facts, link facts and fingerprints. A generated function may request further instances. The build system runs a worklist to a fixed point.

This avoids mutating the declaring module every time a consumer monomorphises a call. New shoots grow beside the trunk. The trunk stays put.

Parallelism and reuse

Canonical identity is assigned before workers race. Canonical merge order applies after workers finish. Semantic, implementation, root, link and documentation changes invalidate different dependents.

The second build of the greeting project should reuse an unchanged greeting artefact and only rebuild what fingerprints say moved.

What can fail here

  • missing fingerprint input or inconsistent identity
  • a generated request that cannot be materialised
  • attempts to publish partial interfaces from diagnosed modules
  • internal artefact failures distinct from source diagnostics

Why Beanstalk does it this way

Alternative: recompile each module for every entry, or let backends inspect mutable frontend state.

Choice: immutable canonical artefacts with explicit lanes and fingerprints.

Benefit: reuse and parallel scheduling preserve one semantic result.

Cost: invalidation and sidecar ownership need precise contracts.

Roadmap and current status

  • Accepted design: immutable CompiledModuleArtifact lanes and fingerprints.
  • Current implementation: progress rows for reuse and generated function support.
  • Queued work: canonical module compilation and future persistent reuse plans.
  • Deferred or outside scope: a backend-owned source cache as authority.
