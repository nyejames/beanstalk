Packaging a module once

Reader promise

  • Understand immutable module artefacts, separate data lanes, fingerprints, generated sidecars and deterministic reuse.

Entry state

  • Validated HIR, borrow facts, folded metadata and static requests.
  • Running checkpoint: place greeting.bst facts into an artefact.

Opening problem

  • Later stages need coherent facts, not a loose mutable AST or repeated source scans.
  • One shared module must serve every entry without entry-specific recompilation.

Section sequence

Four artefact lanes

  • Public semantic interface.
  • Executable HIR and borrow facts.
  • Per-function link facts.
  • Compiler and builder metadata.

Fingerprints and identity

  • Introduce base fingerprints, stable origin identities and export bindings.
  • Put the greeting constant and function in their lanes.

Success, diagnosed and blocked

  • Show success-only public interfaces.
  • Explain diagnosed modules and blocked consumers.

Generated sidecars

  • Show concrete generic functions beside the base module.
  • Explain a project-wide fixed-point request worklist.

Parallelism and reuse

  • Canonical identity before workers.
  • Canonical merge order after workers.
  • Separate semantic, implementation, root, link and documentation changes.

What can fail here

  • Missing fingerprint input, inconsistent identity or a generated request that cannot be materialised.
  • A diagnosed provider cannot expose partial linkable data.
  • Keep source diagnostics distinct from internal artefact failures.

Why Beanstalk does it this way

  • Alternative: recompile each module for every entry or let backends inspect mutable frontend state.
  • Choice: immutable canonical artefacts with explicit lanes and fingerprints.
  • Benefit: reuse and parallel scheduling preserve one semantic result.
  • Cost: invalidation and sidecar ownership need precise contracts.

Roadmap and current status

  • Accepted design: immutable CompiledModuleArtifact lanes and fingerprints.
  • Current implementation: compare progress rows for current reuse and generated function support.
  • Queued work: canonical module compilation and future persistent reuse plans.
  • Deferred or outside scope: a backend-owned source cache.
