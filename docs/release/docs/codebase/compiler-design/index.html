Compiler design: source to page

One chronological route follows a Beanstalk project from source files to generated HTML. The outline marks accepted architecture, current implementation and roadmap gaps for the later writing pass.

From Beanstalk source to a webpage

This series provides educational scaffolding for a later writing pass. It explains compiler concepts through one Beanstalk project. It does not define compiler behaviour or current implementation status.

Reader promise

  • Follow one project from source files to generated HTML.
  • Learn each compiler term when the running example needs it.
  • Track which owner creates each representation.
  • Separate accepted design from current, queued and deferred work.

Running example

Project tree:


    compiler_walkthrough/
    ├── config.bst
    └── src/
        ├── #page.bst
        └── greeting.bst

Keep these source shapes stable across the series:


    -- greeting.bst

    greeting_prefix #= "Hello"

    greet |name String| -> String:
        return [: [greeting_prefix], [name]!]
    ;
    

    -- #page.bst

    import @greeting { greet }

    visitor = "Priya"

    [$html:
        <p>[greet(visitor)]</p>
    ]

Final relevant fragment:


    <p>Hello, Priya!</p>

Pipeline position

Plan one reusable strip with these checkpoints:


    project tree
    -> command and config
    -> canonical source graph
    -> located tokens
    -> prepared shells
    -> bound interfaces and ordered declarations
    -> typed AST and semantic identities
    -> folded constants, generated requests and TIR handoff
    -> structured diagnostics
    -> validated HIR
    -> borrow and ownership side-table facts
    -> immutable module artefacts and fingerprints
    -> entry assembly and reachable function union
    -> deterministic target assignment
    -> JavaScript and Wasm outputs
    -> HTML, runtime assets and output manifest

Outline each article around the current checkpoint. Show both the incoming representation and the outgoing representation.

Owners to introduce

  • Build system: selects commands, discovers source, constructs graphs, schedules modules, plans entries and owns output writing.
  • Compiler frontend and analysis: prepares syntax, binds interfaces, checks semantics, lowers HIR and validates access.
  • Artefact builder: assembles selected compiler facts into an output plan.
  • Target backends: lower validated functions and runtime requirements into JavaScript, Wasm and glue artefacts.

Status language

  • Accepted design: canonical architecture or language contract

  • Queued or active implementation: named roadmap work
  • Deferred or outside scope: intentionally postponed or rejected design
