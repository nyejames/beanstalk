Compiler design / Build-system and frontend boundary

Build-system and frontend boundary

The build system prepares inputs. The frontend compiles them. Keeping the line sharp stops backends from rediscovering source files and the frontend from guessing builder policy.

Build systems provide a BackendBuilder and wrap it in a ProjectBuilder.

The frontend compiles modules up to HIR and borrow validation. The backend builder consumes those compiled modules and emits project artifacts. That split is strict.

Backend builders don't load source files, discover modules, load or parse config.bst, perform semantic frontend compilation or write final output files directly. They declare the build surface the frontend is allowed to see.

BackendBuilder


    pub trait BackendBuilder {
        fn build_backend(
            &self,
            modules: Vec<Module>,
            config: &Config,
            flags: &[Flag],
            string_table: &mut StringTable,
        ) -> Result<Project, CompilerMessages>;

        fn validate_project_config(
            &self,
            config: &Config,
            string_table: &mut StringTable,
        ) -> Result<(), ProjectConfigError>;

        fn frontend_style_directives(&self) -> Vec<StyleDirectiveSpec>;

        fn libraries(&self) -> LibrarySet;
    }

    pub struct ProjectBuilder {
        pub backend: Box<dyn BackendBuilder + Send>,
    }

A backend builder declares these frontend-visible surfaces:

  • source libraries
  • external packages
  • config keys
  • style directives
  • external import providers
  • builder runtime packages
  • supported source file kinds

It receives a validated Config, may validate backend-owned config keys and lowers compiled Module values into artifacts. The frontend doesn't know the final project layout. The backend doesn't know how raw source files were discovered.

LibrarySet

LibrarySet is the build surface exposed to the frontend. It carries typed registries, not loose callbacks or strings.


    pub struct LibrarySet {
        pub external_packages: ExternalPackageRegistry,
        pub source_libraries: SourceLibraryRegistry,
        pub config_keys: ProjectConfigKeyRegistry,
        pub external_import_providers: ExternalImportProviderRegistry,
        pub external_import_cache: ExternalImportProviderCache,
        pub external_import_resolution_table: ExternalImportResolutionTable,
        pub builder_runtime_packages: Vec<BuilderRuntimePackageMetadata>,
        pub source_file_kinds: SourceFileKindRegistry,
    }

ProjectConfigKeyRegistry is declarative Stage 0 metadata. It lists allowed core and backend-owned keys plus value shapes so config.bst can reject unknown declarations, shape-invalid folded values and closed-domain string values before core fields are applied or backend settings are stored.

Config compilation

config.bst is build-system-owned compile-time Beanstalk source. It uses the frontend up to AST so config values can use normal compile-time constants, folded templates, core/builder imports and typed diagnostics.

It isn't compiled as a module. It doesn't produce HIR. It doesn't create runtime start semantics. It doesn't export language-visible declarations.

Authored config entries must be known top-level # constants. Plain top-level bindings are runtime Beanstalk syntax and are rejected in config. Imported constants and support types may participate in config expressions, but imported declarations are support surface only and never become config entries.

Config may contain these support declarations:

  • imports from core or builder libraries
  • type aliases
  • structs
  • choices

Config rejects these by design:

  • project-local imports
  • relative imports
  • mutable bindings
  • functions
  • calls that require runtime behaviour
  • runtime statements
  • local helper constants that are not known config keys
  • traits
  • trait conformances
  • trait incompatibility declarations
  • standalone templates
  • direct top-level const-template page fragments

For example, config rejects this entry-file-only page-fragment form:


    #[$md:
        # Page title
    ]

Config-key constants can use const-record field projection when the expression fully folds. Structured typed config values are still deferred.

External import providers

External import providers are builder-declared hooks. Stage 0 and import preparation use them to turn non-Beanstalk files into typed external package surfaces before AST consumes visibility.

Provider results may also carry registered runtime imports discovered while parsing the external source. That is important for JavaScript imports: the frontend sees a typed package surface, while the backend later emits only the runtime glue and assets that reachable HIR uses.

Provider-capable discovery stays serial for now. It mutates the external package registry, provider cache, resolution table and shared diagnostic string table. Parallel provider-backed discovery needs deterministic provider deltas and ID remapping before it can be safe.

Builder-runtime packages

Builder-runtime package metadata covers builder-owned virtual packages such as @web/canvas. These packages are registered directly in external_packages, then attached to module external-import metadata when reachable HIR references one of their functions.

This lets builder-owned packages share the same runtime asset and glue path as provider-created imports without pretending they were project-local source files.

Source file kinds

Source file kind metadata lets builders opt into source assets that participate in normal source import discovery without becoming Beanstalk modules. HTML registers Beandown .bd and plain Markdown .md content assets this way.

A supported source asset follows the same extensionless source import candidate path as .bst files. Recognised but unsupported source kinds are rejected with typed import diagnostics.

Beandown and plain Markdown become ordinary synthetic content #String constants during header preparation. Beandown nested templates with no explicit directive default to $md during ordinary AST template parsing. They don't get a custom AST node, HIR, borrow-checker or backend path.

Validation lanes

Project-specific config validation stays in BackendBuilder::validate_project_config. It returns ProjectConfigError so normal user config mistakes remain typed CompilerDiagnostic values while infrastructure failures remain explicit CompilerError values.

Complex release optimisations should stay outside the fast frontend path unless correctness requires them. This compiler is built for hot-reload development, so slow project-wide cleverness belongs behind a clear target or release boundary.
