## What modern Wasm looks like in May 2026

Release 3.0 materially expands core Wasm relative to the 1.0 and 2.0 worldview. The current change history folds in extended constant expressions, tail calls, exception handling, multiple memories, 64-bit address spaces for memories and tables, typed or typeful references with `call_ref`, GC types and recursive subtyping, relaxed SIMD, profiles, and custom annotation syntax in the text format. ## What the LIR should be

The core design choice is this: BeanstalkŌĆÖs LIR should be a Wasm-aware optimisation IR, not a literal stack-machine IR. Mature Wasm tooling points in the same direction. Binaryen keeps a main IR that is tree-shaped and optimisation-friendly, then uses an additional Stack IR for stack-machine-shaped optimisation and final code quality work. That is strong evidence that one IR should not be forced to serve both global optimisation and final stack-form emission equally well. ŅłĆciteŅłéturn40view0Ņłü

A good Beanstalk LIR should therefore be a typed CFG-plus-SSA IR with first-class block parameters, explicit terminators, and native tuple or multi-value support. That shape matches current Wasm better than phi-heavy or stack-only forms, because modern Wasm has multi-value functions and blocks, precise reference types, typed `call_ref`, multiple memories, memory64, tags, and GC type graphs. It also lets the backend preserve optimisation quality until a late stackification pass, instead of freezing stack order too early. ŅłĆciteŅłéturn44view0Ņłéturn22view0Ņłéturn20view1Ņłéturn27view1Ņłéturn30view1Ņłü

In concrete terms, the LIR should make these properties explicit rather than implicit:

- **Exact value types.** Model scalar values as `i32`, `i64`, `f32`, `f64`, `v128`, or reference types with explicit nullability and heap type. Do not collapse references to ŌĆ£pointer-likeŌĆØ. `wasm-encoder`ŌĆÖs `ValType` and `RefType` APIs mirror this directly. ŅłĆciteŅłéturn25search0Ņłéturn27view1Ņłü
- **Address and container types.** Memory operations need an explicit memory index, while memory and table declarations need explicit `memory64` or `table64`, sharedness, and, for memories, optional custom page size. Those are not implementation details any more; they are part of the target-visible type system. ŅłĆciteŅłéturn20view1Ņłéturn26view0Ņłéturn27view0Ņłéturn24search1Ņłü
- **Feature requirements on nodes.** A `call_ref`, `throw`, `struct.new`, relaxed SIMD op, atomic op, or continuation op should carry the feature it requires. `wasm-encoder` already exposes instruction families spanning GC, exceptions, atomics, continuations, and even proposed wide-arithmetic operations, so the emitter can only stay sane if legality is decided above it. ŅłĆciteŅłéturn12view0Ņłéturn10search1Ņłü
- **Effect and trap structure.** Loads, stores, conversions, potential traps, allocation, host calls, atomics, and exception edges should be explicit in LIR. Wasm validation and execution rules care about these distinctions, and they matter for correct stackification and local placement. ŅłĆciteŅłéturn36search0Ņłéturn18view3Ņłéturn24search2Ņłü
- **Late local allocation.** SSA values should be primary. Function locals are a late storage decision, especially because modern Wasm has non-defaultable reference types whose valid use depends on initialisation dominance. ŅłĆciteŅłéturn22view0Ņłéturn40view0Ņłü

The most future-proof version of this design is a two-layer backend model: one persistent LIR for analysis and optimisation, plus one derived ŌĆ£Wasm emission planŌĆØ that is structured, stackified, and local-allocated just before passing instructions into `wasm-encoder`. That extra layer is worth it.

## How Beanstalk HIR should translate into that LIR

HIR-to-LIR should resolve every front-end ambiguity that would otherwise leak into instruction selection. That means overload resolution, signedness-sensitive comparisons and conversions, short-circuiting, match lowering, tail position, exception behaviour, and concrete ABI choices must all be fixed before the final Wasm pass. The final emitter should be selecting and ordering instructions, not still deciding language semantics. That separation matters more in Wasm than in many native backends because operand stack shape, block signatures, tag signatures, and reference exactness all depend on earlier semantic choices. ŅłĆciteŅłéturn18view1Ņłéturn18view3Ņłéturn22view0Ņłéturn27view1Ņłü

Expression-oriented HIR should lower into explicit basic blocks with typed block parameters and terminators. That is the cleanest way to map conditionals, loops, short-circuit boolean logic, and pattern-like control flow onto Wasm `block`, `loop`, `if`, `br_if`, and `br_table` structures later. Do not translate HIR directly into a stack of encoder calls. Translate HIR into explicit control flow first; only then structure and stackify it. BinaryenŌĆÖs own split between its main IR and Stack IR exists for exactly this reason. ŅłĆciteŅłéturn40view0Ņłü

Multiple return values should be represented natively in LIR, not faked through hidden out-pointers or synthetic tuple memory slots unless a specific ABI forces that. Multi-value is long since standardised, the 3.0 finished-proposals list still treats it as a core part of the modern language, and the JS embedding explicitly supports multiple values flowing through its host interface. If BeanstalkŌĆÖs HIR already has tuples or multiple result expressions, do not erase them during HIR-to-LIR. ŅłĆciteŅłéturn44view0Ņłéturn45view1Ņłü

Reference-bearing HIR should stay abstract enough to support both a linear-memory runtime and a future GC runtime. The current 3.0 core draft adds managed heap types, reference subtyping, struct and array types, recursive types, and cast or test operations. That is too much surface area to retrofit later if the first LIR hard-wires ŌĆ£all objects are integer addressesŌĆØ. The better approach is to distinguish raw numeric addresses from language references in LIR now, even if the first Wasm backend lowers language references into a linear heap. ŅłĆciteŅłéturn20view2Ņłéturn20view3Ņłü

If Beanstalk eventually wants async functions, generators, or green-thread-like semantics, the first backend should still lower them through explicit state-machine or runtime-assisted transformations in HIR or early LIR rather than depending on stack switching. Stack switching is active at phase 3, not standardised, even though `wasm-encoder` already has continuation-related instructions in its instruction model. That makes it a good future target and a bad initial dependency. ŅłĆciteŅłéturn43view0Ņłéturn24search0Ņłéturn10search1Ņłü

## How LIR should lower to Wasm

Lowering from LIR to Wasm should be organised as a fixed sequence of backend passes, not a monolithic ŌĆ£emit WasmŌĆØ step. The sequence I would use is: target-feature selection, Wasm type and ABI materialisation, CFG structuring, stackification, local allocation and copy coalescing, section planning, then encoder emission. That order matches both the binary formatŌĆÖs strict section layout and the fact that some metadata, such as data counts and branch hints, can only be emitted correctly once layout facts are known. ŅłĆciteŅłéturn13view0Ņłéturn36search0Ņłéturn17view1Ņłü

The structuring pass should turn CFG edges into Wasm nesting, preserving block signatures and branch values. Tail-position calls should be marked so the final pass can choose `return_call` or `return_call_indirect` when the feature is enabled and the call is legal. Exception-enabled code should become explicit tag declarations plus `try_table`, `throw`, and related control flow rather than pretending that exceptions are just host callbacks. Multiple memories and memory64 should already be solved at the LIR level, because memory and table instruction operand types depend on the declared address type of the subject memory or table. ŅłĆciteŅłéturn18view1Ņłéturn18view3Ņłéturn20view1Ņłéturn31view0Ņłéturn32view0Ņłü

Section planning deserves its own backend data structure. The core binary grammar orders non-custom sections as type, import, function, table, memory, tag, global, export, start, element, optional data count, code, then data, and `wasm-encoder::Module::section` deliberately leaves correct ordering to the caller. A Beanstalk backend should therefore build a section plan before emission, not append opportunistically. That same plan should know whether a data-count section is required; the core spec says it must be present if any data index appears in code, and the encoder exposes `DataCountSection` explicitly. ŅłĆciteŅłéturn36search0Ņłéturn13view0Ņłéturn35view0Ņłü

One subtle but important pass is stackification. This is where SSA values become an evaluation order that minimises `local.set` and `local.get` churn while still preserving precise semantics. The pass should prefer stackable trees, keep multi-value edges intact where profitable, and spill only when duplication, control-flow joins, or validation constraints require it. This is the place to exploit Wasm-specific code-shape wins; it should not contaminate the main LIR. BinaryenŌĆÖs separate Stack IR is a useful conceptual model here. ŅłĆciteŅłéturn40view0Ņłü

Finally, if you expose a deterministic backend mode, implement it here as a legality and selection policy. In that mode, relaxed SIMD should either be avoided or mapped only where the deterministic profile still permits fixed behaviour, and floating-point operations should be treated with deterministic-profile expectations about NaN production. The specŌĆÖs profile design makes this a backend option, not a language redesign. ŅłĆciteŅłéturn38view0Ņłéturn37search2Ņłü

## How to use wasm-encoder effectively

The current `wasm-encoder` crate, in the ŅłĆentityŅłé["organization","Bytecode Alliance","open source foundation"]Ņłü wasm-tools ecosystem, exposes `Module` as the central builder, section-specific encoders for the core binary format, optional `wasmparser` integration, and component-model APIs behind a feature flag. The crate root currently lists version 0.248.0 and complete documentation coverage. That is a good fit for Beanstalk because the library is intentionally close to the Wasm binary model rather than trying to hide it behind a high-level IR. ŅłĆciteŅłéturn8view0Ņłü

Use `TypeSection` and its single-use `CoreTypeEncoder` as the boundary between BeanstalkŌĆÖs type world and WasmŌĆÖs type section. The encoder can write plain function signatures as well as array, struct, subtype, continuation, and recursive type-group entries. If BeanstalkŌĆÖs runtime starts linear-memory-first, this still matters: the type path should be architected so GC and recursive types can be added without changing the emitter API shape later. ŅłĆciteŅłéturn11search0Ņłéturn30view1Ņłéturn29view0Ņłéturn30view0Ņłü

Use `Function` and `CodeSection` in two modes. For ordinary emission, create grouped locals with `Function::new`, write instructions through `instructions()` or `instruction()`, and append the body with `CodeSection::function`. For caching or incremental compilation, use `Function::into_raw_body` and later inject the saved bytes with `CodeSection::raw`; the docs explicitly call out this combination as suitable for caching code-generation results. That is valuable for a compiler like Beanstalk, where repeated incremental builds are likely. ŅłĆciteŅłéturn10search0Ņłéturn13view1Ņłéturn14view0Ņłü

Treat `InstructionSink` as the final mnemonic layer, not as the backend itself. Its API surface is instruction-shaped and very broad, covering scalar numeric ops, SIMD, atomics, GC instructions, tag and exception machinery, continuation instructions, and more. Once BeanstalkŌĆÖs late emission plan has made types, immediates, indices, offsets, alignments, and feature legality explicit, final emission should mostly be a mechanical walk that calls the corresponding sink methods. That is exactly where `wasm-encoder` is strongest. ŅłĆciteŅłéturn12view0Ņłéturn41search1Ņłü

Use the optional `wasmparser` integration aggressively in tests. The crate root example already validates emitted bytes with `wasmparser::validate`, and the `reencode::Reencode` API can parse whole modules, sections, function bodies, type groups, imports, exports, data, and tags while also providing a hook for interspersing or detecting sections. That makes it ideal for round-trip tests, differential tests, canonicalisation passes, fuzz minimisers, and ŌĆ£emit, parse, re-emitŌĆØ sanity checks. ŅłĆciteŅłéturn8view0Ņłéturn41search0Ņłü

Emit auxiliary sections deliberately. `NameSection` helps debugging and has an ordering discipline for subsections; `ProducersSection` is a widely supported non-standard convention with recommended fields such as `language`, `processed-by`, and `sdk`; and `BranchHints` encodes the `metadata.code.branch_hint` custom section using byte offsets within function bodies. That last detail matters: branch-probability metadata in Beanstalk should stay symbolic until the final instruction layout is known, then be translated into branch-hint offsets late. ŅłĆciteŅłéturn16view0Ņłéturn17view0Ņłéturn17view1Ņłü

Keep component support out of the first core backend. `wasm-encoder` already includes `Component`, `ComponentBuilder`, canonical-function support, and other component-model types, but the proposals repository still lists Component Model at phase 1. The right architecture is to generate a strong core Wasm module first and keep the ABI boundary clean enough that a component wrapper can be added later as packaging rather than as the backendŌĆÖs core responsibility.
